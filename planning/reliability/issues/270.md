# Issue #270 — fix(ios): Apple live transcription clears text after speech pause

## Status
**Approved** — 2026-04-08

## Summary
iOS bug: `commitIfImplicitReset` heuristic in `iOSLiveTranscriber` has threshold gaps that allow text loss on pause.

## Reliability assessment
- **Blast radius**: iOS only, Apple transcription engine only. No macOS impact.
- **Rollback**: TestFlight manual dispatch; App Store review acts as natural gate. Regression = don't promote TestFlight build.
- **Failure modes**:
  - Short-text miss: `lastFormattedString.count >= 10` threshold drops commits for short utterances before pause
  - Gradual reset miss: `currentText.count < lastFormattedString.count / 2` (50%) may not catch incremental resets
  - Task cancellation race: `restartRecognitionTask` cancels old task but `isShuttingDownRecognitionTask=true` guards the error callback; seems safe but worth verifying in implementation
  - Wrong-fix risk: over-eager commit → duplicate text; under-commit → text still lost
- **Audio tap correctness**: Tap closure reads `self?.recognitionRequest` (not captured); reassignment to `newRequest` in restart is correct — existing tap buffers route to new request automatically.

## Guardrails documented
1. Address short-text threshold edge case (< 10 chars)
2. Address gradual-reset edge case (50% threshold)
3. Verify restart cancellation race in implementation PR
4. Add Sentry breadcrumb / os_log when implicit reset fires (operational visibility)

## Approval rationale
Low operational risk: iOS-only, TestFlight gating, no new infrastructure, extends existing pattern. Worst-case regression is duplicate text or continued text loss — neither catastrophic.
