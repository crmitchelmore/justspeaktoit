# Issue #149 — Portable planning-memory pattern after signed-commit correction

**Status:** needs re-review
**Last updated:** 2026-03-25

## Summary
The issue was reopened as a portability decision for repos where required signed commits block `planning/*` branch writes.

## Verified facts
- The live throwaway-branch test showed Contents API writes land as `verified: false`, `reason: unsigned` here.
- Branch exemption and workflow commit signing are the remaining options that satisfy signed-commit enforcement without changing the product behaviour.
- A dedicated memory repo is still possible, but only justified when multiple repos share the same planning team.

## Performance stance
- Do **not** recommend Contents API.
- Branch exemption has the lowest runtime cost, but it depends on per-repo governance and is not the portable default.
- Workflow commit signing has a one-time setup cost but near-zero steady-state overhead and keeps the existing memory topology intact.
- A dedicated memory repo adds more operational surface and should be reserved for multi-repo shared teams.

## Approval status
Not yet. Re-evaluate once Security and Architecture converge on branch exemption versus workflow commit signing.
