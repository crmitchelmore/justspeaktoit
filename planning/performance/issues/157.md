# Issue #157 — Add live capture health details to the HUD

## Status
`planning:performance-approved` — approved.

## Summary
Add a lightweight capture-health view to the HUD: mic permission, input device, provider, and a recent-latency signal.

## Verified repo facts
- HUDManager is push-based with `@Published` properties; no polling loop.
- `TranscriptionManager` tracks session timing but no latency bucket is published to HUDManager yet.
- `PermissionsManager` already exists; mic permission state can be observed.
- Issue constraints explicitly state: no aggressive polling, no noisy HUD updates.

## Performance guardrails (for implementation)
1. Rate-limit latency signal updates — only update on state transitions or at most ~1 Hz.
2. Device enumeration must be notification/KVO-driven, not polled.
3. Latency signal should be a last-value bucket (e.g. <1s / 1-3s / >3s), not a live average to avoid churn.

## Approval
Approved. The issue's explicit constraint "do not poll aggressively or add noisy HUD updates" plus push-based HUDManager architecture means performance risk is low. Implementation should enforce update-rate guardrails noted above.

## Cross-role
Only `planning:needs-quality` remains. Architecture and Security approved.
