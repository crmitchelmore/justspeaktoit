# Issue #149 — Agentic workflow live retest + portable memory topology

**Status:** needs-architecture (re-engaged on raised bar)
**Date:** 2026-03-25

## Summary

Maintainer explicitly asked Architecture to argue for a portable default topology for `planning/*` repo-memory when orgs enforce signed-commit rulesets. Prior approval was on the weaker "this is just workflow validation" basis; new bar is "concrete standard pattern other repos can adopt."

## Architectural position

**Recommended portable default: Contents API writes to `planning/*` branches.**

- Replace `git CLI push` with GitHub REST API (`PUT /repos/.../contents/...`) for all memory writes
- Contents API commits are GitHub-verified by default; satisfies signed-commit rulesets without branch exemptions
- Preserves existing `planning/*` branch topology; no new repos or secrets needed
- Minimal blast radius: only the memory-write step in each role workflow changes

**Upgrade path for multi-repo orgs: dedicated `{org}/planning-memory` repo.**

- Only warranted when ≥3 code repos share the same planning team pattern
- Isolates memory from app source; avoids touching each code repo's branch rules
- Requires cross-repo write token (added complexity)

**Decision rule:** Switch the persistence mechanism before the topology.

## Open questions / blockers

1. Does the team agree Contents API is the right default, or does a dedicated repo serve the single-repo case better?
2. Security: any concern with GitHub-signed commits vs. user-signed commits for audit purposes?

## Cross-role dependencies

- Security: branch exemption vs. signed-workflow-commits vs. API-based path — Security's input needed before we finalise
- Code Quality: owns the "minimum documentation + workflow contract" for portability
