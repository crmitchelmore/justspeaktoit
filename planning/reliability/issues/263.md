# Issue #263 — Perf Improver: O(1) incremental stats and WAL write fix in HistoryManager

**Status**: Reliability approved (with guardrail)
**Date**: 2026-04-08

## Proposal
Two changes in `HistoryManager` (`@MainActor final class`):
1. **WAL fix**: Remove redundant disk re-read in `appendToWAL`. Use `pendingWrites` directly since it's already the canonical in-memory WAL state after a successful `replayWAL()` on startup.
2. **Stats fix**: Replace O(n) `calculateStatistics(for: allItemsOnDisk)` calls in `append`/`update`/`remove` with the existing-but-unused O(1) `updateStatisticsForAppend/Update/Remove` helpers. All three guard against `nil` cachedStatistics with a fallback.
3. **Bug fix**: `update()` now captures `oldItem` before replacing (was a pre-existing correctness issue in `updateStatisticsForUpdate`).

## Reliability assessment
- **Blast radius**: HistoryManager display/history data only. No audio/transcription logic touched.
- **Rollback**: Standard mac-v* re-tag → Sparkle serves prior version. Straightforward.
- **Stats risk**: Low. Guard in all three `updateStatisticsFor*` methods falls back to full recompute when `cachedStatistics == nil`. Drift bounded by guard.
- **WAL risk (narrow)**: If `replayWAL()` fails due to a storage write IO error (not decode error), the WAL file persists with valid entries but `pendingWrites = []`. Proposed fix would then overwrite the WAL with just the new entry, silently losing the old entries. Current code preserved them via disk re-read. This is a real but narrow failure mode (requires IO error on storage write at startup + subsequent write) — already a degraded state logged to Sentry. Both code paths lose entries in the decode-error scenario.
- **Implementation guardrail needed**: `appendToWAL` should check `pendingWrites.isEmpty && FileManager.default.fileExists(at: walURL)` at startup and fall back to disk read in that case. OR track whether `replayWAL` succeeded.

## Key code facts
- `replayWAL()` on success: merges WAL into storage, removes WAL file. On failure: logs error, returns `[]` (WAL file persists).
- `flushImmediately()` writes full `allItemsOnDisk` snapshot, then clears `pendingWrites` and WAL.
- Pre-existing: `flushImmediatelySync` uses `DispatchSemaphore.wait()` on `@MainActor` with `Task.detached` calling `await flushImmediately()` — the MainActor deadlock pattern. Flagged as pre-existing, not introduced by this change.
