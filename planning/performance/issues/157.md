# Issue #157 — Add live capture health details to the HUD

## Status
`planning:performance-approved` — maintainer proposal resolved all open questions.

## Summary
Add a lightweight capture-health view to the HUD: mic permission, input device, provider, and a latency signal.

## Verified repo facts
- HUDManager is push-based; updates come from callers, not a polling loop.
- `AudioInputDeviceManager` is CoreAudio property-listener-driven (`kAudioHardwarePropertyDevices`, `kAudioHardwarePropertyDefaultInputDevice`) — not polled.
- `LatencyBadge.swift` / `ModelCatalog.LatencyTier` exposes a static per-model tier (`.instant/.fast/.medium/.slow`).
- `PermissionsManager.statuses` is `@Published` and observable.

## Resolved questions
1. **Latency signal**: Use static `ModelCatalog.LatencyTier` for v1. Zero measurement overhead. If actual per-session latency is added in v2, the pattern is rolling average of `TranscriptionResult.duration`, throttled push on final segment.
2. **Update-rate cap**: Confirmed — only on state transitions from `PermissionsManager`, `AudioInputDeviceManager`, `AppSettings.liveTranscriptionModel`. Never on audio ticks or per-word events.
3. **Device enumeration**: `AudioInputDeviceManager` already uses CoreAudio property listeners — no polling risk.

## Approval
Approved. Resolved by maintainer comment at #4122655250.
