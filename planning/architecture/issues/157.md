# Issue #157 — Add live capture health details to the HUD

**Status:** needs-architecture (first pass)
**Date:** 2026-03-25

## Summary

Extend the macOS HUD to surface microphone permission, active input device, transcription provider, and a latency signal.

## Key repo facts verified

- `HUDManager.Snapshot` (struct, SpeakApp) is the existing aggregated display type — the right extension point.
- All data sources are already in SpeakApp (not SpeakCore): `PermissionsManager`, `AudioInputDeviceManager`, `AppSettings` (liveTranscriptionModel), `TranscriptionProviderRegistry`, `ModelCatalog.LatencyTier`.
- `LatencyBadge.swift` already exists for rendering latency tier — reuse it.
- `MainManager` already drives all HUD updates and holds (via AppEnvironment) all required managers.
- No `CaptureHealthStatus` type exists anywhere.

## Open questions / blockers

1. **Update trigger**: Does health state refresh only at session start (safe, cheap) or live throughout recording (needs care to avoid noisy updates)?
2. **Latency signal**: Confirmed static `LatencyTier` from `ModelCatalog` is safe and reusable. Open question is whether last-round-trip elapsed (already in `snapshot.elapsed`) is additionally useful.

## Architecture recommendation

- Add a `CaptureHealth` nested struct to `HUDManager.Snapshot` (macOS-only; no SpeakCore involvement needed).
- `MainManager` subscribes to `PermissionsManager`, `AudioInputDeviceManager`, and `AppSettings` Combine publishers and calls a new `updateCaptureHealth(...)` on `HUDManager` — follows the existing pattern.
- Do NOT create a SpeakCore type for this; all sources are macOS-only.
- Use `LatencyTier` (static, already in ModelCatalog) as the latency signal; avoid runtime sampling.

## Approval notes

Pending clarification on update trigger (session-start vs. live). Once confirmed, plan is architecturally sound.
