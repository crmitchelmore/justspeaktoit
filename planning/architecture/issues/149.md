# Issue #149 — Portable planning-memory pattern after signed-commit correction

**Status:** needs re-review
**Last updated:** 2026-03-25

## Summary
The issue started as a workflow-health retest and later became a portability-design decision. A live repo test disproved the earlier assumption that GitHub Contents API writes satisfy signed-commit enforcement here.

## Verified facts
- A throwaway-branch test using `PUT /repos/{owner}/{repo}/contents/{path}` produced `verified: false`, `reason: unsigned`.
- `planning/*` branch memory works in this repo today because these branches are not blocked by required signed commits.
- `Docs/agentic-workflows.md` now documents the two strict-repo paths that actually satisfy the constraint: exempt `planning/*` or configure workflow commit signing.

## Architecture stance
- Do **not** recommend Contents API as the signed-commit-safe default.
- Keep the existing `planning/*` branch topology; change the signing setup before changing the topology.
- For a single repo with strict rules, prefer workflow commit signing as the portable default because it preserves the existing topology across repos.
- A branch exemption is an acceptable repo-local fallback when governance allows it.
- A dedicated `{org}/planning-memory` repo is only worth the extra complexity when multiple code repos share the same planning team.

## Cross-role dependency
- Security should confirm that workflow commit signing is the acceptable default under signed-commit governance.

## Approval status
Not yet. Prior approval was invalidated by the live test disproving the Contents API assumption.
