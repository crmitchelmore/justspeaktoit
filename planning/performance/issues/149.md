# Issue #149 — Portable planning-memory pattern after signed-commit correction

**Status:** performance-approved
**Last updated:** 2026-03-25

## Summary
Workflow verification issue that evolved into a real portability decision for repos where signed-commit rulesets block `planning/*` branch writes.

## Verified facts
- The live throwaway-branch test showed Contents API writes land as `verified: false`, `reason: unsigned` here. Contents API is NOT a signed-commit-safe default.
- Branch exemption and workflow commit signing are the remaining viable options.
- A dedicated memory repo is only justified when multiple repos share the same planning team.

## Performance stance
- **Do not recommend Contents API** — ruled out by live evidence.
- Branch exemption has the lowest runtime cost but depends on per-repo governance; not a portable default.
- Workflow commit signing: one-time setup, near-zero steady-state overhead (~1-3s per run), portable default.
- Dedicated memory repo: disproportionate for single-repo teams; reserved for multi-repo shared teams.
- Both viable options are acceptable from a cost perspective.

## Approval status
**Approved.** No application runtime risk. Memory write costs for both viable options are well within budget.
Prior blocker (cross-role convergence) was pending, but maintainer correction is now in memory and both options are cost-acceptable. Approved on explicit maintainer re-review request.
