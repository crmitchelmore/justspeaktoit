# Issue #157 — Add live capture health details to the HUD

**Status:** architecture-approved, ready-for-dev
**Date:** 2026-03-25

## Summary

Extend the macOS HUD to surface microphone permission, active input device, transcription provider, and a latency signal.

## Key repo facts verified

- `HUDManager.Snapshot` (struct, SpeakApp) is the existing aggregated display type — the right extension point.
- All data sources are already in SpeakApp (not SpeakCore): `PermissionsManager`, `AudioInputDeviceManager`, `AppSettings` (liveTranscriptionModel), `TranscriptionProviderRegistry`, `ModelCatalog.LatencyTier`.
- `LatencyBadge.swift` already exists for rendering latency tier — reuse it.
- `MainManager` already drives all HUD updates and holds (via AppEnvironment) all required managers.
- No `CaptureHealthStatus` type exists anywhere.

## Agreed implementation boundary (from Code Quality comment)

- `CaptureHealthSnapshot` is a plain struct in `SpeakApp` (aligned with architecture recommendation)
- Updated via explicit `HUDManager.updateCaptureHealth(_:)` fed by `MainManager` bindings
- No separate observable monitor object
- No reads from high-frequency audio-level or per-word transcription paths
- Provider/error state mapped to fixed categorical values before reaching the snapshot

## Architecture approval rationale

- All data sources macOS-only → no SpeakCore involvement; correct module boundary
- Extension via nested/companion struct on HUDManager.Snapshot follows existing pattern
- MainManager drives updates — consistent with its orchestration role
- Static LatencyTier avoids runtime sampling overhead
- Update trigger: session-start or binding-driven, not hot-path polling
