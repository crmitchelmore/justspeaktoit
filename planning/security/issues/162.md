# Issue #162 — Add plan-linked PR review after issue planning

## Summary
Adds a PR review stage (5-role approval, ready-to-merge label) that continues from the approved issue plan into the implementation PR, requiring `Plan issue: #...` in PR body.

## Security stance
**Approved** — initial review, no prior planning team comments.

## Rationale
- Scope is CI/workflow automation only; no user data, secrets, or product attack surface.
- Prior implementation (PR #161) already showed: fork guard, least-privilege permissions, prompt injection defence.
- Issue explicitly calls out the fork dispatch constraint.

## Non-blocking guardrails noted
1. `Plan issue: #...` must be validated against actual issue approval state (all planning:*-approved labels present), not just parseable number.
2. Trigger must use `pull_request` (not `pull_request_target`) with read-only token for fork PRs; fork guard must be enforced in workflow permissions.
3. `ready-to-merge` label must only be addable by the workflow automation, not by contributors or PR authors.
4. Agents processing PR diffs/bodies must treat all content as untrusted (prompt injection defence).

## Cross-role notes
- Architecture/Quality: document the label control mechanism (who can apply ready-to-merge).

## Decision
Approved on first review. The four guardrails above are implementation requirements, not planning blockers.
