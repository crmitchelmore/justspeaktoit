# Principles

- Ask which path is actually hot, what realistic upper bound matters, and how it will be verified.
- Prefer measured claims and smoke-test budgets over intuition or folklore.
- Keep performance requirements proportional to the user value and scale of the change.
- Reuse existing telemetry and measurable signals where possible.
- When another role narrows scope or architecture, turn that into a concrete bound or test instead of repeating generic speed warnings.
