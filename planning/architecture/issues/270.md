# Issue #270 — fix(ios): Apple live transcription clears text after speech pause

## Status
Architecture: **approved** (initial review, Planning Kickoff + Product comment only so far)

## Summary
Bug in `iOSLiveTranscriber.swift`: transcribed text is cleared when Apple's SFSpeechRecognizer times out on silence. The codebase already has a committed-text accumulation pattern (`committedText`, `commitIfImplicitReset`, `restartRecognitionTask`) but two gaps allow text loss.

## Structural analysis
- Fix is correctly scoped to `Sources/SpeakiOS/Services/iOSLiveTranscriber.swift` — iOS-only, no cross-module changes.
- Two gap areas:
  1. **Error path** — `handleRecognitionResult` returns without committing `lastFormattedString` when a non-shutdown error arrives; if Apple fires an error (silence timeout) instead of or before `isFinal`, text is lost.
  2. **Heuristic threshold** — `commitIfImplicitReset` requires `lastFormattedString.count >= 10`; short utterances before a pause are not protected.
- No new seams, no interface changes to `stop()` / `onFinalResult` shape needed.

## Guardrails
- Commit must happen before any early return in the error path (unless `isShuttingDownRecognitionTask` is true — intentional cancel).
- On-device vs server recognition may have different timeout behaviors; fix should work for both paths controlled by `preferOnDevice`.
- Do not change `stop()` semantics or `onFinalResult` shape.
