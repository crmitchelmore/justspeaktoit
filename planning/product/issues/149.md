## Summary
Workflow-health check reframed by maintainer as a real portability-design decision: what is the standard memory-persistence pattern for repos with signed-commit requirements?

## Key facts verified
- `Docs/agentic-workflows.md` already documents two viable paths: (1) exempt `planning/*` branches from signed-commit requirements, (2) configure workflow commit signing upfront
- Contents API writes produce `verified: false, reason: unsigned` commits — NOT a valid signed-commit solution (proven by live test, maintainer comment 4122680947)
- The `planning/*` memory branches work fine in this repo today (no signed-commit enforcement on those branches)

## Product stance
- Product value is clear: persistent role memory improves cross-issue planning quality
- The docs already point to the two correct paths (exemption or signing)
- Contents API is ruled out — do not recommend it as a portable default
- Ephemeral memory is acceptable as last-resort degraded mode, must be documented explicitly as such
- Product approves; Architecture should confirm whether exemption or workflow-signing is the canonical recommendation for other repos
