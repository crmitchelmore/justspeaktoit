# Issue #157 — Add live capture health details to the HUD

**Status:** architecture-approved (re-approved 2026-04-10 after /doit cycle)
**Date:** 2026-04-10

## Summary

Extend the macOS HUD to surface microphone permission, active input device, transcription provider, and a latency signal.

## Key repo facts verified

- `HUDManager.Snapshot` (struct, SpeakApp) is the existing aggregated display type — the right extension point.
- All data sources are already in SpeakApp (not SpeakCore): `PermissionsManager`, `AudioInputDeviceManager`, `AppSettings` (liveTranscriptionModel), `TranscriptionProviderRegistry`, `ModelCatalog.LatencyTier`.
- `LatencyBadge.swift` already exists for rendering latency tier — reuse it.
- `MainManager` already drives all HUD updates and holds (via AppEnvironment) all required managers.

## Agreed implementation boundary (intact from first round, confirmed in /doit cycle)

- `CaptureHealthSnapshot` is a plain struct in `SpeakApp` (no SpeakCore involvement)
- Updated via explicit `HUDManager.updateCaptureHealth(_:)` fed by `MainManager` bindings
- No separate observable monitor object
- No reads from high-frequency audio-level or per-word transcription paths
- Three explicit update triggers: `PermissionsManager.statuses`, `AudioInputDeviceManager`, `AppSettings.liveTranscriptionModel`
- Provider/error state mapped to fixed categorical values before reaching the snapshot
- Static `LatencyTier` for v1
- Plain struct value semantics: no closures, no @ObservableObject, no back-references to source managers

## Cross-role tension flagged (view-layer only, non-architectural)

Product (Alex) says: show health panel in idle/ready state.
Design (Riley, original approval) said: show only in recording and failure phases.
View-layer rendering condition — no structural implication. PR must record the chosen phase set and have Product + Design confirm at review time.

## Architecture approval rationale

- All data sources macOS-only: no SpeakCore involvement; correct module boundary
- Extension via CaptureHealthSnapshot companion struct follows existing pattern
- MainManager drives updates: consistent with orchestration role
- Static LatencyTier: zero overhead
- Plain struct: one-directional push, easy to test with literal values
