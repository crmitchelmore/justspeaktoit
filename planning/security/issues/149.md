# Issue #149 — Portable planning-memory pattern after signed-commit correction

**Status:** needs re-review
**Last updated:** 2026-03-25

## Summary
The workflow-health retest evolved into a security-relevant operations question: how should role memory persist when strict repos require signed commits on `planning/*` branches?

## Verified facts
- The maintainer's live test showed Contents API writes produce `verified: false`, `reason: unsigned` commits here.
- Persistent role memory remains inside repo boundaries if either workflow commit signing or an explicit `planning/*` branch exemption is used.
- A dedicated memory repo widens the credential and repository boundary, so it should not be the default for a single repo.

## Security stance
- Do **not** approve Contents API as a signed-commit-safe default.
- Acceptable strict-repo paths are:
  1. workflow commit signing with a dedicated automation key/secret,
  2. an explicit `planning/*` branch exemption approved by repo or org admins.
- Dedicated memory repo is only justified if the shared-team value clearly outweighs the broader trust boundary.
