# Issue #149 — Agentic workflow live retest after secret refresh

**Status:** approved
**Date:** 2026-03-25

## Summary

Initially a pure workflow retest with no app performance risk. Re-opened as a design debate about repo-memory persistence portability (signed-commit rulesets blocking `planning/*` branch pushes).

## Assessment

### Original approval
No runtime code path or UX latency is changing. Approved immediately.

### New design question (comment 4122656618)
Maintainer asked Performance to call out workflow cost if memory persistence becomes more complex. Three options evaluated:

1. **Branch exemptions**: Zero cost change. Requires org-level governance. Not portable.
2. **GitHub API signed commits** (tree/blob/commit API): +1-3s per write per agent run. No branch exemption needed. Fully portable. Preferred default.
3. **External stores** (Gist, artifact, external DB): Adds external service dependency, higher latency, operational complexity. Not proportionate.

Key guardrail: unbounded memory file growth is the primary performance risk regardless of mechanism. Cap `recent-decisions.md` (10 entries max), prune closed issue files.

## Decision

Approved. Performance stance unchanged. Recommended standard pattern: GitHub API signed commits as portable fallback, with memory file size caps enforced.
