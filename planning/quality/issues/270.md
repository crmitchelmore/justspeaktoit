# Issue #270 — fix(ios): Apple live transcription clears text after speech pause

**Status:** needs-quality (first pass)
**Labels:** planning:in-discussion, needs-product, needs-security, needs-performance, needs-quality, needs-architecture
**Created:** 2026-04-08

## Summary
iOS Apple (SFSpeechRecognizer) transcription resets accumulated text after a pause.
File in scope: `Sources/SpeakiOS/Services/iOSLiveTranscriber.swift`

## Existing mechanisms
- `committedText` — accumulates text committed on isFinal
- `commitIfImplicitReset()` — detects reset via length heuristic (>= 10 chars, < half previous)
- `restartRecognitionTask()` — restarts after isFinal signal

## Known failure modes identified by Casey
1. **Threshold miss**: `commitIfImplicitReset` only triggers if text >= 10 chars AND drops by > 50%. Short text and gradual resets slip through.
2. **Error path data loss**: `handleRecognitionResult` error branch bails without committing `latestResult`. If Apple sends silence-timeout error with partial results, those results are lost.
3. **Silent task termination without isFinal**: If task dies without any callback, restart never fires.

## Quality blockers
1. Which failure scenario does the fix target? All three, or just the threshold miss?
2. **No SpeakiOSTests SwiftPM target** — `commitIfImplicitReset` logic cannot be unit-tested via `make test`. Verification story needs to be explicit: either extract to SpeakCore for unit testing, or specify concrete device-log verification steps.
3. Does the error path commit `latestResult` before bailing?

## Cross-role dependency
- Product (Alex) confirmed scope: iOS only, fix in iOSLiveTranscriber, don't change stop() semantics or onFinalResult shape.

## Latest stance
Not yet approved. Waiting on: (1) concrete implementation direction specifying which failure modes are addressed, (2) verification story given no iOS SwiftPM test target.
